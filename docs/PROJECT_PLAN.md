## 1. 프로젝트 개요

### 2-2. 기술 스택

| 영역 | 기술 | 선정 이유 |
|---|---|---|
| **런타임** | Node.js 20+ | 비동기 I/O 최적화, npm 생태계 |
| **프레임워크** | Express.js | 경량, MCP SSE 지원에 적합 |
| **데이터베이스** | MongoDB Atlas (Mongoose) | 토큰 영구 저장, 서버리스 환경 적합 |
| **캐싱** | In-Memory (Map) | 단일 서버 환경에서 외부 의존성 없이 충분 |
| **인증** | OAuth 2.0 (Cafe24) | 카페24 공식 인증 스펙 |
| **배포** | Render | GitHub 연동 자동 배포, 무료 티어 |
| **AI 통신** | MCP (Model Context Protocol) | AI 에이전트 표준 프로토콜 |

## 3. 핵심 기능 명세

### 3-1. MCP Tool #1 — 맞춤형 상품 추천 (`search_cafe24_real_products`)

**트리거 예시**: "건성인데 좋은 크림 추천해줘", "민감성 피부에 맞는 선크림 있어?"

| 항목 | 내용 |
|---|---|
| **입력 파라미터** | `skin_type`, `concerns[]`, `category`, `count` |
| **데이터 소스** | Cafe24 Admin API → 전체 상품 100개 조회 |
| **핵심 알고리즘** | AI 채점 알고리즘 (가중치 기반) |
| **출력 형태** | 콤팩트 가로 테이블 (Pre-rendered Markdown) |

#### 채점 알고리즘 상세

```
[채점 공식]
최종 점수 = 카테고리 매칭(+3) + 피부타입 매칭(+2) + 고민별 매칭(+2 each) + 랜덤 Jitter(0~0.4)

[동의어 사전 매칭]
사용자 입력 "건성" → 검색 확장: [건성, 보습, 수분, 촉촉, 세라마이드, 장벽, 드라이, 너리싱, 모이스처, 하이루론]
사용자 입력 "크림" → 검색 확장: [크림, 보습크림, 모이스처라이져, 수분크림, 너리싱, 날크림, 나이트크림, 겔크림]

[중복 방지 로직]
1. 정규식으로 [1+1], (증정), 기획, 세트 등 수식어 제거
2. 핵심 본명(Base Name) 추출
3. 동일 본명 상품은 대표 1개만 노출
4. 나머지는 upsell_options 배열에 묶어서 연관 제안

[랜덤 가산점 조건]
- 맞춤형 추천 (키워드 매칭된 경우): Jitter 적용 → 매번 다른 순위
- 범용 질문 (매칭 없는 경우): Jitter 미적용 → 공식 진열 순서 유지
```

### 3-2. MCP Tool #2 — 베스트셀러 랭킹 (`get_bestseller_ranking`)

**트리거 예시**: "잘 팔리는 거 뭐야?", "베스트 5위까지 보여줘", "인기 순위"

| 항목 | 내용 |
|---|---|
| **입력 파라미터** | `count` (기본 5, 최대 10) |
| **데이터 소스** | Cafe24 Admin API → 베스트 카테고리(No.47) 진열 순서 |
| **핵심 특징** | 쇼핑몰 관리자가 설정한 공식 랭킹을 그대로 반환 |
| **신뢰도** | 매번 동일한 순위 보장 (랜덤 개입 없음) |

#### 데이터 흐름

```
1. 카테고리 47번 상품 목록 조회 → product_no 배열 획득
2. product_no 배열로 상품 상세 정보 일괄 조회
3. 카테고리 진열 순서 유지하며 상세 정보 매핑
4. 콤팩트 가로 테이블 + 썸네일 이미지 Pre-render
5. AI에게 "이 순서 절대 변경 금지" 지시와 함께 전달
```

### 3-3. 토큰 자동 갱신 (Auto-Refresh)

```
[흐름도]
API 호출 → 401 에러 감지
          → Refresh Token으로 새 Access Token 발급 요청
          → 발급 성공 → MongoDB에 새 토큰 저장
          → 원래 API 요청 자동 재시도
          → 사용자는 아무것도 모른 채 정상 응답 수신
```

| 항목 | 내용 |
|---|---|
| **Access Token 수명** | 2시간 |
| **Refresh Token 수명** | 14일 |
| **저장소** | MongoDB Atlas (서버 재시작 시에도 유지) |
| **복구 불가 시** | 브라우저에서 `/cafe24/start` 수동 재인증 안내 |

## 4. UI/UX 설계

### 4-1. 출력 레이아웃 전략

| 상황 | 레이아웃 | 이유 |
|---|---|---|
| **맞춤 추천** (3개) | 가로 테이블 + 이미지 | 한눈에 비교 가능, 구매 결정 도움 |
| **베스트 랭킹** (5~10개) | 가로 테이블 + 이미지 | 공식 순위 한눈에 파악 |

### 4-2. Pre-render 전략 (AI 레이아웃 통제)

**문제**: AI(LLM)가 서버가 지시한 마크다운 표 양식을 무시하고 자의적으로 레이아웃을 변경하는 현상 발생

**해결**: 백엔드 서버(Node.js)에서 마크다운을 100% 완성하여 AI에게 전달하고, "단 한 글자도 수정하지 말고 그대로 복사하라"는 강제 지시

```
[Before] AI에게 "이런 형식으로 그려줘" 라고 규칙 전달 → AI가 멋대로 변경
[After]  서버가 완성된 마크다운 텍스트를 만들어서 AI에게 "복사만 해" → 일관된 출력
```

### 4-3. AI 페르소나

| 항목 | 설정 |
|---|---|
| **역할** | 셀퓨전씨 전속 뷰티 컨설턴트 |
| **말투** | 친근하고 전문적인 언니/오빠 같은 상담 톤 |
| **분석 스타일** | 성분, 제형, 사용감 등 구체적 포인트를 짚어주는 전문 분석 |
| **제한** | 상품당 최대 3줄, 장황한 설명 금지 |

## 6. 기술적 의사결정 기록 (ADR)

### ADR-001: 토큰 저장소를 인메모리에서 MongoDB로 전환

- **상황**: Render 무료 티어는 서버가 주기적으로 재시작됨 → 인메모리 토큰 유실
- **결정**: MongoDB Atlas 클라우드를 토큰 영구 저장소로 채택
- **이점**: 서버 재시작/재배포 시에도 토큰 유지, 수동 재인증 빈도 최소화

### ADR-002: Rate Limit 도입 후 철거

- **상황**: `express-rate-limit` 도입 → AI 백엔드 통신과 간섭하여 정상 요청도 차단
- **결정**: Rate Limit 일시 철거, 추후 AI 백엔드 IP 화이트리스트 기반으로 재도입 예정
- **교훈**: AI 에이전트 통신은 일반 사용자 요청과 다른 패턴을 가짐

### ADR-003: 맞춤 추천과 랭킹을 별도 MCP Tool로 분리

- **상황**: 단일 도구로 모든 질문을 처리 → "베스트셀러" 질문에도 AI 채점 알고리즘이 개입하여 매번 순위가 변동
- **결정**: `search_cafe24_real_products` (맞춤 추천)과 `get_bestseller_ranking` (공식 랭킹)을 별도 도구로 분리
- **이점**: 맞춤 추천은 다양성(Jitter), 랭킹은 신뢰도(고정 순위) 각각 보장

### ADR-004: 뷰티 전문 동의어 사전 도입

- **상황**: "건성 피부에 좋은 크림" 질문 → 상품명에 "건성"이란 글자가 없어 매칭 실패 → 전부 "공식 추천 상품"으로 표시
- **결정**: 뷰티 전문 동의어/유사어 사전(Synonym Dictionary) 구축
- **예시**: "건성" → [보습, 수분, 촉촉, 세라마이드, 장벽, 드라이, 너리싱, 모이스처, 하이루론]
- **이점**: 키워드 매칭률 비약적 향상, 추천 이유 구체화

### ADR-005: AI 레이아웃 통제를 위한 Pre-render 전략

- **상황**: AI에게 "가로형 표로 그려줘" 지시 → AI가 자의적으로 세로형/줄글로 변환
- **결정**: 백엔드에서 마크다운 텍스트를 100% 완성하여 AI에게 "복사만 하라" 지시
- **이점**: 모든 환경에서 일관된 UI/UX 보장
